(翻译) golang版本历史

发表于 2020-06-14

背景：内部使用的golang版本比较低，准备升级到新的版本。那么如何选择一个新的版本呢？是最新的？还是最新的低一个版本？或者其他？理所当然的的感觉是比最新第一个版本。但是为什么了？翻译下golang的发布历史，看看每个版本到底做了什么

Go 1.14 的介绍

自1.13发布后，历经6个月，1.14发布了。该版本的大部分变动是工具链、运行时和库。当然也保证了对旧版本的兼容信。我们期望所有的go程序能继续编译和运行。

go命令的模块功能已经可以在生产环境使用了，我们鼓励所有的用户迁移到go模块依赖管理。如果你遇到问题，不能迁移，请保证对应有一个开发issue（如果不再1.15的里程碑里，请让我们知道为什么阻止你迁移，我们会适当有限考虑这个问题）

语言的改动

对重叠接口的建议，1.14现在允许在重叠模式下内嵌接口。内嵌的接口的方法和嵌入的接口有相同的名字和相同的标识符。这个解决了钻石形嵌入的问题。显示声明的方法还是和原来一样是唯一的

工具

Go 命令

Vendoring

如果go.mod文件指定1.14或者更高的版本，go命令缺省的设置-mod=vendor. 新的值 -mod=mod 会让go命令从cache中加载模块。

在-mod=vendor情况下，go命令会校验vendor/modules.txt和go.mod的一致性。

go list -m 不再静默省略不再vendor目录下的依赖。如果设置了-mod=vendor，但是信息不在vendor/modules.txt里面，会显示的报错，

Flags

go get 不再接受-mod命令。以来，这个flag的设置会被忽略或者导致编译失败。

当go.mod是只读的，并且vendor目录没有时，-mod=readonly 是缺省设置。

-modcacherw 是新的选项。允许go命令保留模块的默认权限。这样可以添加不存在的文件，以便测试。然而，这样也允许了直接通过rm -rf删除模块缓存

-modfile=file 是新的文件指导go命令读取可选的go.mod，而不是模块根目录下的文件。但是go.mod文件必须还是要存在在模块根目录，虽然不会使用。当指定了-modfile，go.sum也会使用对应目录下的文件。

环境变量

GOINSECURE 是新的环境变量，允许go命令拉取模块时，不校验证书。是逗号分隔的模式列表

模块外的命令

当设置了module-aware模式，很多模块命令的功能会受限。例如，go build，go run和其他构建命令只能编译标准库和命令行指定的go文件

以前go命令会解决每个包到最新的版本，但是不会记录路径和或者版本号。这样会导致编译缓慢不可再生的编译问题。

go get还是像以前一样工作，所以给go mod download 和 go list -m 一个显示的版本。

+incompatible 版本

如果一个模块最新的版本包含一个go.mod文件，go get不会更新到一个不兼容的主版本，除非显式要求。当拉取模块时，go list也会忽略不兼容的主版本，除非是通过代理报告的。

go.mod 文件的维护

如果一个间接依赖的版本已经被隐式被其他主模块依赖，那么除了go mod tidy 外，其他的go命令不再会删除require目录

如果只是改变装饰，除了go mo tidy,其他的go命令不会修改go.mod文件。

当-mod=readonly被指定，那么go命令不会因为缺少go指令或者一个错误注释而失败

模块下载

go命令支持子版本仓库。

go命令支持模块代理和其他http访问的明文错误信息。一个错误仅仅在满足是有效utf8编码和只包含图形化字符和空格是才会显示

测试

go test -v现在会立刻打印 t.Log,而不是等到测试结束。

运行时

提升了defer 的大部分使用场景下的性能。直接使用延迟函数几乎是零开销。现在可以在性能核心代码中使用defer而不用关心性能影响。

协程现在是异步调度。所以没有函数调用的循环不会导致潜在的调度器死锁或者导致明显的垃圾回收延迟。除了一下平台都支持：wondows/arm,darwin/arm,js/wasm,和plan9/*

在Unix系统，包括Linux和macOs，抢占的后果是，用1.14编译的程序会收到更多的信号。意味着使用了syscall和golang.org/x/sys/unix的程序会看到更多的EINTR错误。这些程序不得不处理这些错误。比如重试。可以参考man 7 signal或者相似的文档查看更多信息。

页分配器会更高效，在高GOMAXPROCS下显著的减少锁竞争。在并行申请大内存，会有更低的延迟、更高吞吐量。

time.After,time.Tick,net.Conn,SetDeadline 使用的内部的时间和friends，更高效，更低的锁竞争，更少的上下文切换。这是用户透明的性能提升。

编译器

添加-d=checkptr编译时选项，添加指引检查代码是否符合unsafe.Pointer安全规则。在-race和-mscan选项下是默认配置，可以通过-gcflags=al=-d=checkptr=0关闭。-d=checkptr具体检查如下：

  1. 当通过unsafe.Pointer转换到*T的时候，结果必须适当的分配给T。
  2. 如果算数指针结果指向一个栈对象，那么unsafe.Pointer类型的操作数之一必须指向同一个元素。

-d=checkptr不推荐在windows使用，会导致标准库报告失败。

通过-json选项，编译器能报告键优化后的机器能识别的log，包括行，逃逸分析，边检检查，空检查。

重新支持了详细的逃逸分析诊断。以前的版本废弃了。

macOs下所有go的符号带上了下划线，满足平台约定。

对于fuzzing，这个版本包括了实验性的支持编译插入覆盖仪表。详细信息参考issue 14565.这个api在未来的版本可能变化。

边界检查消除现在使用分片的创建的信息，可以消除检查小于int的类型索引。

核心库

新byte序列hash包

1.14引入了一个新的包hash/maphash，提供了bytes序列的hash函数。这些函数的目的是被用在hash tables或者其他需要将字符串或者byte序列映射到无符号的64位整型的均匀分布的数据结构中。

这些函数是耐撞的，但是不是密码安全的。

哈希值在一个程序里面是一致的，但是不同进程是不同的。

库的主要变化

一如既往，这些变化是兼容旧版本的。

crypto/tls

crypto/x509

debug/dwarf

encoding/asn1

encoding/json

Decoder支持新方法InputOffset返回当前解码的偏移量。

Compact不再转移 U+2028和U+2029字符。这些不是文档功能。要正确的转移，查看HTMLEscape

Number不再接受无效的数字，这样更接近文档化的行为。如果程序需要接受无效数字，比如空字符串，考虑封装Unmarshaleer

go/build

go/doc

io/ioutil

log

math

math/big

math/bits

mime

mime/multipart

net/http

新的Header和Values可以获取到所有规范键的值。

新的Transport成员DialTLSContext能用来指定一个可选的函数为非代理的HTTPS请求建立TLS连接。这个新的成员代替DialTLS。DialTLS将继续工作，但是新的代码应该考虑使用DialTLSContext，允许不再使用时，立刻关闭连接。

在windows平台，ServeFile可以支持大于2GB的文件

net/http/httptest

net/textproto

net/Unmarshaleer

当分析URL失败，’Error’会报告无法解析的URL.这样提供了清理结构，和其他分析错误保持一致。

os/signal

plugin

reflect

StructOf支持通过设置PkgPath字段创建未导出字段。

runtime

runtime.Goexit不再被 panic/recover 中断。

runtime/pprof

strconv

sync

释放一把高竞争的锁，会直接将CPU调度到写一个等待锁的协程。这个提升了多核CPU机器下，高竞争态锁的性能。

testing

text/template

unicode

unicode更新到了Unicode 12.0。
