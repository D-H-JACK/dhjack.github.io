  1. 选择一个施加了负载的应用程序，执行下面这些任务（其中的一些需要用到动态跟踪）：
  • 在开始如何测量前，你预计这个应用程序是CPU密集型的还是I/O密集型的？说一下你的道理

难道一个程序只能是二选一么?不能同时是CPU密集型和I/O密集型？如果要预计，应该是先看架构图或者说流程图。看主要的逻辑是计算某个结果，还是偏重于传输数据。还是说用top命令看cup的占用率和io wait的比例?从某种角度看，是一个相对的概念。同样的程序，同样的负载，不同的机器配置上，可能对cpu的占用和I/O的占比都不太一样。其实可以量化处理，就单独看一条数据流，会执行多少cpu指令，会传输多少磁盘数据，网络数据。

  • 如果是CPU密集型的（I/O密集型的），确定要用到的观察工具。

ltrace对golang不起作用，因为golang是静态编译

sudo perf top -e sched:sched_switch
sudo perf top -e branch-misses
sudo perf record -e ‘cycles’ – myapplication arg1 arg2
sudo perf report
sudo perf stat

strace -o output.txt -T -tt -e trace=all -p 28979
strace -f -F -o ~/dcop-strace.txt dcopserver
strace -f -o configure-strace.txt -e execve ./configure
strace -ff -o task php nginx
strace -cp
strace -T -vvvv -d -Ff -s 128 -o strace.log -p PID
strace -v -T -d -Ff -c -p

strace底层使用内核的ptrace特性来实现其功能。它能够打开应用进程的这个黑盒，通过系统调用的线索，告诉你进程大概在干嘛。
